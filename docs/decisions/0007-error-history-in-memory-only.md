# 0007. 에러 이력: 메모리 전용 저장

## 상태
확정됨

## 배경 (Context)
중복 판정용 해시셋과 리포트용 누적 에러 목록을 어디에 저장할지 결정이 필요했다. 파일/DB로 영속화하면 프로그램 재시작 후에도 이력이 유지되지만 구현이 복잡해지고, 프로젝트별로 이력을 분리해야 하는 문제(어느 프로젝트의 이력인지 구분)가 추가로 발생한다.

## 결정 (Decision)
에러 이력은 메모리에만 유지한다. 프로그램을 재시작하면 이력은 초기화된다.

## 근거 / 검토했던 대안 (Rationale / Alternatives)
- loglens는 개발 세션 동안 계속 켜두는 상주 프로세스로 사용될 것으로 예상된다(데모 앱은 수시로 재시작되지만 loglens 자체는 거의 재시작되지 않음).
- 이 데이터가 유실되어도 치명적이지 않다: 원본 로그 파일은 그대로 남아있고, loglens가 만드는 건 "분석 결과 캐시"에 가깝다. 재시작 후 같은 에러가 다시 발생하면 한 번 더 AI에 물어보는 정도의 비용만 발생한다.
- 규모도 문제가 되지 않는다: 에러 이력은 "예외타입+위치" 문자열과 짧은 분석 결과 텍스트 수준이라, 하루 종일 켜놔도 많아야 수백~수천 건(수 MB) 수준이다.
- 파일 영속화를 검토했으나, 영속화 시 프로젝트(watch-dir)별로 이력 파일을 분리해야 "아무 터미널에서 리포트를 실행했을 때 엉뚱한 프로젝트 이력이 섞이는" 문제를 피할 수 있어 구현 복잡도가 크게 늘어난다. 메모리 전용 + 같은 프로세스 명령([0008](0008-report-generation-trigger.md))으로 이 문제 자체를 구조적으로 회피했다.

## 결과 (Consequences)
- `ErrorDeduplicator`와 `ErrorStore`는 둘 다 순수 인메모리 자료구조(`Set`, `List`)로 구현한다.
- 장기간 누적 리포트(예: 여러 날에 걸친 이력)가 필요해지면 이 결정을 재검토해야 한다.

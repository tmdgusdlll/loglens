# 0012. 정상 종료 시 shutdown hook으로 리포트 자동 생성

## 상태
확정됨

## 배경 (Context)
에러 이력은 메모리 전용으로만 유지되고([0007](0007-error-history-in-memory-only.md)), 리포트는 개발자가 그 터미널에서 `r` + Enter를 직접 입력해야만 생성된다([0008](0008-report-generation-trigger.md)). 그런데 개발자가 `r`을 한 번도 누르지 않은 채 세션을 끝내면(Ctrl+C 등), 메모리에 쌓여 있던 그 세션의 에러 이력 전체가 아무 흔적 없이 사라진다. "깜빡하고 안 눌렀다"는 흔한 실수로 전체 세션 데이터를 잃는 건 사용자 경험상 아쉬운 지점이다.

## 결정 (Decision)
`Runtime.getRuntime().addShutdownHook(...)`으로 JVM 정상 종료 시점에 그때까지의 `ErrorStore` 전체를 리포트 파일로 한 번 더 생성한다. 파일 경로는 세션 시작 시 고정된 것과 동일한 `reportPath`([0011](0011-report-file-history-preserved.md))를 그대로 재사용하므로, 수동으로 `r`을 여러 번 눌렀든 안 눌렀든 마지막 상태가 항상 그 파일에 반영된다.

## 근거 / 검토했던 대안 (Rationale / Alternatives)
- **에러 발생 시마다 즉시 리포트에 반영(완전 자동화)**: 검토했으나 기각. 같은 에러가 반복 발생하는 경우까지 매번 파일을 다시 쓰게 되면 무한루프성 버그 상황에서 쓰기 빈도가 급증하고, 감시 루프(ADR 0002)와의 결합도가 올라가며, 강제종료 대비 원자적 쓰기(임시파일 + `ATOMIC_MOVE`)까지 필요해져 복잡도가 크게 늘어난다.
- **DB/즉시 durable 쓰기로 강제종료(SIGKILL, 전원 손실)까지 대비**: 검토했으나 지금 스코프에서는 오버엔지니어링으로 판단해 채택하지 않음. loglens는 로컬 개발 중 임시로 켜두는 보조 도구이지 영속적 기록 시스템이 아니며, ADR 0007이 이미 "메모리 전용 + 프로젝트별 이력 분리 문제 회피"를 이유로 파일/DB 영속화를 기각한 바 있다. **추후 개선 과제로 남겨둔다** — 만약 강제종료 상황에서도 데이터 유실을 없애야 하는 요구가 실제로 생기면, 그때 DB(SQLite 등) 또는 에러 감지 즉시 durable 쓰기 방식을 다시 검토한다.
- **shutdown hook(채택)**: 정상 종료(Ctrl+C, `System.exit()`, main 정상 반환) 케이스만 커버하면 되므로 세션당 딱 한 번만 추가로 쓰면 충분하고, 기존 `r` + Enter 콜백과 동일한 `Runnable`을 재사용해 코드 추가가 최소화된다.

## 결과 (Consequences)
- `kill -9` 등 강제 종료나 크래시 상황에서는 hook 자체가 실행되지 않으므로, 그 순간까지의 이력은 여전히 유실될 수 있다 — 이건 알려진 한계이며 위 "추후 개선 과제"로 명시적으로 남겨둔다.
- `main()`의 `generateReport` 콜백이 `Runnable`로 추출되어 `StdinCommandListener`와 shutdown hook 양쪽에서 재사용된다.

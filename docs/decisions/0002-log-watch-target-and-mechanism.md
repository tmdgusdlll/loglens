# 0002. 감시 대상 지정 방식 & 파일 감시 메커니즘

## 상태
확정됨

## 배경 (Context)
loglens가 어떤 로그 파일을 감시할지 사용자가 지정하는 방법과, 파일 변경을 감지하는 내부 메커니즘을 정해야 했다. loglens와 데모 앱은 별도 프로세스로 동시에 떠 있으며, 데모 앱이 실시간으로 로그를 쌓는 동안 loglens가 그 파일을 계속 지켜보는 구조다.

## 결정 (Decision)
- **지정 방식**: `--watch-dir=<데모 앱 루트 경로>`로 프로젝트 디렉토리를 지정한다. loglens는 그 안의 `logs/*.log`를 자동 탐색하며, 여러 개면 가장 최근 수정된 파일을 활성 로그로 선택한다. 파일이 아직 없으면(앱 시작 전) 생성될 때까지 대기한다.
- **감시 메커니즘**: 직접 폴링 방식을 사용한다. 300ms 주기로 파일 크기를 확인하고, 커진 경우에만 `RandomAccessFile`로 마지막으로 읽은 오프셋부터 seek해서 새로 추가된 바이트만 읽는다.

## 근거 / 검토했던 대안 (Rationale / Alternatives)
**지정 방식**
- 로그 파일 경로 직접 지정: 가장 단순하고 명확하지만, 매번 정확한 파일 경로를 알아야 해서 사용 편의성이 떨어짐.
- 디렉토리 지정 + 자동 탐색(채택): 사용 편의성이 좋고 Spring Boot의 `logs/` 관례에 자연스럽게 맞음.

**감시 메커니즘**
- 직접 폴링(채택): 구현 단순, OS/파일시스템 상관없이 동작 일관적, 지연시간을 폴링 주기로 예측 가능하게 조절 가능.
- Java NIO `WatchService`: 이벤트 기반이라 이론적으로 효율적이지만, **macOS에서는 JDK 기본 구현이 내부적으로 10초 폴링으로 동작**해 "즉시 분석"이라는 핵심 요구사항과 맞지 않음. 새 파일 생성(로그 로테이션) 감지도 별도 처리가 필요.
- 외부 프로세스 `tail -F` 실행: 구현은 쉽지만 외부 프로세스 의존이 생겨 "프레임워크 없이 순수 자바"라는 프로젝트 취지에 어긋나고 Windows 호환이 안 됨.

파일 크기 변화 감지 자체는 항상 신뢰할 수 있다 — 파일에 텍스트가 append되면(단 한 줄이라도) 그 바이트 수만큼 파일 크기가 즉시 늘어나는 것이 파일시스템의 기본 동작이며, Logback의 파일 아웃풋은 줄 단위로 즉시 flush되므로 로그가 찍히자마자 반영된다.

## 결과 (Consequences)
- 로그 파일 로테이션(예: 자정마다 새 파일 생성)에 대한 처리는 현재 범위 밖이며, 데모 앱이 개발 중 짧게 실행되는 시나리오를 가정한다.
- 폴링 주기(300ms)는 튜닝 가능한 값으로 남겨두되 기본값으로 고정한다.

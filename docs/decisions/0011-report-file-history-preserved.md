# 0011. 리포트 파일명에 타임스탬프를 넣어 과거 리포트 보존

## 상태
확정됨

## 배경 (Context)
에러 이력은 메모리 전용으로만 유지한다([0007](0007-error-history-in-memory-only.md)). 그런데 리포트는 항상 같은 파일명(`loglens-report.html`)에 덮어써지도록 구현되어 있어서([0008](0008-report-generation-trigger.md)), loglens 프로세스를 재시작한 뒤 다시 `r`을 눌러 리포트를 생성하면 그 시점의 (재시작 이후) 이력만 담긴 파일이 이전 리포트를 덮어써 버린다. 개발자가 재시작 전에 참고하던 과거 에러 분석 내용을 다시 볼 방법이 없어지는 문제가 있었다.

## 결정 (Decision)
리포트 파일명에 **loglens 프로세스가 시작된 시각**의 타임스탬프를 한 번만 부여한다: `reports/loglens-report-{yyyyMMdd_HHmmss}.html`. 같은 세션(프로세스) 안에서 `r` + Enter를 여러 번 눌러도 이 파일 하나에 계속 덮어쓴다 — `ErrorStore`의 생명주기(프로세스 범위, [0007](0007-error-history-in-memory-only.md))와 리포트 파일의 생명주기가 그대로 맞아떨어진다. loglens를 재시작하면 새 타임스탬프로 새 파일이 생성되고, 이전 세션의 파일은 그대로 `reports/`에 남는다.

## 근거 / 검토했던 대안 (Rationale / Alternatives)
- **에러 이력을 DB/파일로 영속화**: [0007](0007-error-history-in-memory-only.md)이 이미 검토하고 기각한 방향과 동일한 문제(프로젝트별 이력 분리, 구현 복잡도 증가)를 다시 떠안는다. 이번 요구는 "과거 리포트를 참고하고 싶다"는 것이지 "메모리 이력 자체를 영속화해야 한다"는 것이 아니므로 과한 해법이다.
- **매 생성마다 새 파일**: 처음 검토했으나, `r`을 여러 번 누르는 동안(같은 세션 안)에도 계속 새 파일이 쌓여 `reports/` 폴더가 금방 지저분해진다. 세션 하나 = 리포트 파일 하나로 묶는 게 사용자가 원하는 바에 더 맞다.
- **세션 시작 시각으로 고정한 파일명(채택)**: `ErrorStore`는 여전히 세션(프로세스) 범위의 인메모리 캐시로 남겨두고([0007] 그대로 유지), 세션 하나당 리포트 파일 하나만 생성/누적 갱신한다. 코드 변경이 `Main`에서 파일 경로를 한 번만 계산해 재사용하도록 하는 것으로 국한된다.

## 결과 (Consequences)
- `reports/` 폴더에는 loglens를 재시작한 횟수만큼(세션 수만큼) HTML 파일이 남는다 — 오래된 파일을 정리(삭제/보관 정책)하는 기능은 이번 결정 범위 밖이며, 필요해지면 별도로 논의한다.
- [0008](0008-report-generation-trigger.md)에 명시된 고정 파일명(`loglens-report.html`)은 더 이상 유효하지 않다. [0008]의 상태를 "대체됨"으로 갱신했다.
- 같은 세션 안에서 시스템 시각을 되돌리는 등 비정상적인 상황이 아니라면 파일명 충돌은 없다(세션 시작 시 1회만 타임스탬프를 계산하므로 초 단위 충돌 우려도 사라진다).

# 0003. 중복 에러 판정 기준

## 상태
확정됨

## 배경 (Context)
같은 에러가 반복 발생할 때마다 AI 분석을 다시 호출하면 무료 API 할당량을 낭비하고 터미널이 도배된다. "같은 에러"로 판단할 기준이 필요했다.

## 결정 (Decision)
**예외 타입 + 발생 위치(스택트레이스 첫 번째 `at ...` 줄)**를 조합해 해시화하고, 이 해시가 이미 존재하면 중복으로 판정해 AI 재호출을 건너뛴다.

## 근거 / 검토했던 대안 (Rationale / Alternatives)
- 예외 타입만: 구현은 가장 간단하지만, 서로 다른 위치에서 발생한 동일 예외 타입을 하나로 묶어버려 중요한 새 에러를 놓칠 수 있음.
- 예외 타입 + 발생 위치(채택): 같은 코드 위치에서 같은 예외가 반복되면 중복으로 보고, 다른 위치의 같은 예외는 별개로 취급. 오탐/누락 균형이 가장 좋음.
- 전체 스택트레이스 전문 해시: 가장 엄격하지만, 라이브러리 버전이나 라인번호 미세한 차이로도 "다른 에러"로 오판단될 위험이 큼.

## 결과 (Consequences)
- 중복 판정은 `ErrorDeduplicator`가 AI 호출 **이전 단계**에서 수행하므로, 중복 에러는 API 호출 자체가 발생하지 않는다.
- 중복으로 판정된 에러도 `ErrorStore`에는 발생 횟수로 카운트되어 리포트에 "N회 반복 발생"으로 표시된다.

# 0008. HTML 리포트 생성 트리거: 같은 프로세스 stdin 명령

## 상태
대체됨(→ [0011](0011-report-file-history-preserved.md))

## 배경 (Context)
에러 이력을 메모리 전용으로 저장하기로 했으므로([0007](0007-error-history-in-memory-only.md)), 누적 리포트를 별도 프로세스(`java -jar loglens.jar --report`)로 실행하면 빈 상태에서 시작해 아무 내용도 못 만든다. 리포트를 언제, 어떻게 트리거할지 재정의가 필요했다.

## 결정 (Decision)
리포트는 별도 프로세스가 아니라, **감시가 진행 중인 그 loglens 프로세스**에 대고 트리거한다. 감시 루프가 도는 동안 별도 스레드로 표준입력(stdin)을 함께 듣고 있다가, 사용자가 그 터미널에 `r`을 입력하고 Enter를 치면 그 자리에서 메모리 상의 `ErrorStore`를 읽어 HTML 파일(`loglens-report.html`)을 생성한다.

## 근거 / 검토했던 대안 (Rationale / Alternatives)
- **별도 프로세스로 `--report` 실행**: 메모리 전용 저장 결정과 근본적으로 충돌한다. 별도 프로세스는 빈 `ErrorStore`로 시작하므로 아무것도 만들 수 없다. 이를 해결하려면 결국 파일 영속화가 필요해지고, "어느 프로젝트의 이력인지" 구분 문제까지 다시 떠안게 된다.
- **같은 프로세스 stdin 명령(채택)**: 별도 프로세스/통신 없이 메모리 저장과 자연스럽게 맞아떨어진다. "어느 프로젝트인지" 문제 자체가 발생하지 않는다 — 그 프로세스는 애초에 하나의 `--watch-dir`만 감시하고 있기 때문이다.

## 결과 (Consequences)
- 메인 실행 루프는 파일 폴링과 stdin 리스닝을 동시에 처리해야 한다(별도 스레드 또는 non-blocking 체크).
- 리포트는 loglens를 실행 중인 그 터미널에서만 생성할 수 있다. 이는 의도된 제약이다.
